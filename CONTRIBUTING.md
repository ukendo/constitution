# Як пропонувати зміни

Цей репозиторій містить **вимоги до статуту**. Зміни в `main` приймаються лише через pull request.

## Правила

1. У одному PR можна змінювати кілька вимог — як зручніше для роботи.
2. Зовнішні учасники — через **fork** і PR. Write-доступ до репозиторію не потрібен.
3. Обговорення без готового тексту — у [Discussions](https://github.com/ukendo/constitution/discussions) або Issue з міткою `обговорення`.
4. Для Issue / PR використовуйте мітки (`вимога`, `редакція`, `юридичне`, `суперечність` тощо); у описі варто вказати номери вимог (В-N).

## Порядок роботи

```text
1. Створіть гілку від актуального main
2. Змініть потрібні файли в vymohy/<тема>/V-NN.md (і README, якщо змінюється короткий опис)
3. Перевірте сумісність з іншими вимогами (див. «Суперечності» нижче)
4. Відкрийте PR з короткою назвою (напр. «В-7, В-8: …» або «редакція: …»)
5. У описі PR поясніть *чому* змінюєте (не лише що); озвучте знайдені суперечності
6. Дочекайтеся рев’ю та merge
```

Після merge гілка видаляється автоматично.

## Суперечності між вимогами

Якщо зміна **суперечить** уже існуючій вимозі (або залишає розрив / невизначеність) — **не мовчати**.

### Що зробити в тексті вимог

1. У **обох** (або всіх залучених) файлах додати блок з однаковою семантикою:

   ```markdown
   > ⚠️ **Суперечність з [В-N](шлях/до/V-NN.md):** коротко, у чому розрив. **Обговорити:** варіанти вирішення.
   ```

   Для м’яких конфліктів / відкритих питань процедури можна:

   ```markdown
   > ⚠️ **Узгодити з [В-N](шлях/до/V-NN.md):** …
   ```

2. Посилання мають бути **відносними** і робочими з поточної папки теми (`../інша-тема/V-NN.md` або `V-NN.md` у тій самій темі).
3. Оновити таблицю **[Відкриті суперечності](README.md#відкриті-суперечності)** у `README.md` (додати рядок або прибрати, якщо конфлікт знято).
4. У описі PR / Issue **явно назвати** конфлікт і номери В-N; за потреби — мітка `суперечність`.

### Чого уникати

- Виправляти «свою» вимогу так, щоб інша лишилась несумісною **без** позначки.
- Позначати суперечність лише з одного боку (завжди — двосторонньо / на всі залучені файли).
- «Вирішувати» суперечність мовчки в PR без згадки в описі — рев’юер має побачити відкрите питання.

Індекс відкритих конфліктів: [README → Відкриті суперечності](README.md#відкриті-суперечності).

## Мітки Issues / PR

| Мітка | Навіщо |
|-------|--------|
| `вимога` | Зміна вимог (В-N) |
| `юридичне` | Потрібна юридична перевірка |
| `редакція` | Редакційні правки |
| `обговорення` | Питання без готового рішення |
| `суперечність` | Є або з’явився конфлікт між вимогами |

## Версії

Зрізи фіксуються git-тегами (`v0.1`, `v0.2`, …) і згадуються в [CHANGELOG.md](CHANGELOG.md).
Повний знімок вимог для архіву: [`Вимоги_до_статуту_нової_федерації_v0.1.md`](Вимоги_до_статуту_нової_федерації_v0.1.md).

При розбіжностях між зібраним документом і файлами `vymohy/<тема>/V-NN.md` пріоритет у **окремих файлів вимог**.
